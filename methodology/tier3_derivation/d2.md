# D2 — Structural Analysis (Tier 3, scoped)

## Re-confirmation of scope

Same as D1(a)–(c). This document analyzes hierarchy-level atom activation and leaf access patterns for the fused tcgen05 FP8 MMA kernel that replaces `dequant_gather_batched_kernel` + `torch::bmm`. All other phases of `launch_topk_c` remain Tier 2.

## Step 1a — Dimension fate (file 06)

| Dim | Size | Fate | Notes |
|---|---|---|---|
| B (batch) | variable (≤ 32 in workload dist.) | AREA | Independent per CTA; maps to grid.y |
| N (seq pos, rows of scores) | N_max = max_num_pages × 64 | AREA | Independent per N-tile; maps to grid.x |
| H (query heads) | 64 | AREA (for MMA output rows) / REDUCE (in downstream weighted_relu_sum, unchanged) | 64 equals MMA_M exactly — one MMA issue covers all heads for one N-tile |
| D (head-dim) | 128 | REDUCE | Collapsed inside MMA via SRG×FXP (tiled as 4 slabs of 32) |

New-to-Tier-3: **D becomes an explicit REDUCE dim driven by a tile loop of static count 4** (K=32 per slab for kind::f8f6f4, file 05's FXP row).

GATE dimension (top-K over N) is NOT inside this kernel's scope — it runs after the kernel exits. Gate molecule unchanged.

### SRG vs MON classification for the new reductions
- D-reduction inside MMA: **SRG** (sum over k is associative + commutative + IDE=0 + DIS). Implemented as a free parallel tree at the warp level, but delegated to tensor-core hardware — the semiring classification authorizes this hardware delegation (file 03, SRG row: "tensor core MMA for dot products" is the warp-level GPU manifestation of SRG).
- K-slab outer accumulation (4 slabs into one TMEM cell): **SRG × FXP = Online Algorithm** (file 03 Matrix11×5, SRG×FXP row). Hardware implements this via `enable_input_d=true` after slab 0 (TMEM acts as the register-carry state).

## Step 1b — Access patterns (file 09, leaf types)

### Leaf 1: Q global → SMEM
- **Depth: 0** (arithmetic leaf).
- Address: `addr(b, h, k) = q_base + b*64*128 + h*128 + k`.
- Dtype: FP8 e4m3 (1 byte/elt).
- Total bytes per CTA: `64 × 128 = 8192 B` (one copy per batch element).
- Coalescing: within a warp, lane stride along k = 1 → stride-1 bytes → vectorize as `uint4` (16 bytes per thread) → 8 bytes × 16 threads per row = `uint4 × (128/16)=8 threads per row`. With 4 warps cooperating: 64 threads × 16 bytes × 8 rows per pass = 8192 bytes in one cooperative pass. **Coalesced.**
- TMA-eligibility: row stride = 128 B = multiple of 16. **TMA permitted.** But D3 will choose cooperative load (no benefit from TMA for a single 8 KB tensor that is reused N_tile-many times from SMEM).

### Leaf 2: K global → SMEM (page-indexed — depth-1 lookup chain)
- **Depth: 1** (lookup leaf; file 09 Depth-1 row — "paged KV cache").
- Hop 1: `phys = block_table[b, page_id]` — 4 B int32 lookup.
- Hop 2 (final): `k_data_addr = k_cache_base + phys * PAGE_BYTES + tok * 128 + k`.
- Scale gather (parallel final leaf): `scale_addr = k_cache_base + phys * PAGE_BYTES + PAGE_DATA_BYTES + tok * 4`.
- Dtype (data): FP8 e4m3 (1 B/elt). Dtype (scale): FP32 (4 B/elt).
- Per-page bytes loaded: 64 × 128 = 8192 B FP8 + 64 × 4 = 256 B scale = 8448 B (exactly `PAGE_BYTES`).
- Coalescing: within one row (one token), lane stride along k = 1 → stride-1 bytes → vectorize as `uint4` (one warp loads 128 B = one row in one instruction across 16 lanes).
- Row stride = 132 B — **NOT a multiple of 16 → TMA forbidden** (GT-9 invariant re-confirmed).
- Null/bounds predicate: Hop 1 — `page_id < num_pages_for_seq[b]`; Hop 2 — `tok < seq_lens[b]`. Both reduce to the unified condition `global_tok = page_id*64 + tok < seq_lens[b]`. Zero-fill on fail (preserves GT-26 mask chain).

**Important:** `num_pages_for_seq[b] = ceil(seq_lens[b]/64)`. A single CTA processes one N-tile of size MMA_N. That N-tile's token range is `[n_tile*MMA_N, (n_tile+1)*MMA_N)`. Pages straddled by the CTA's N-tile are `page_id ∈ [n_tile*MMA_N/64, (n_tile+1)*MMA_N/64)`. For MMA_N=64 → exactly one page per CTA (no straddling). For MMA_N=128 → two pages. For MMA_N=256 → four pages. **D3 tile-shape decision must minimize block_table traffic.**

### Leaf 3: Scale scratchpad → SMEM → register (per-column multiplier)
- **Depth: 0** after the scale is in SMEM (D2 treats SMEM-resident values as arithmetic leaves).
- Address: `scale_smem[n_in_tile]` — one FP32 per N column within the tile.
- Usage: post-MMA, per thread, multiplied into FP32 accumulator loaded from TMEM before storage to GMEM.

### Leaf 4: scores_all SMEM → GMEM
- **Depth: 0**.
- Address: `addr(b, h, t) = scores_base + b*64*N_max + h*N_max + t`.
- Dtype: FP32 (4 B/elt).
- Coalescing: within a warp, lane stride along t = 1 → stride-1 floats → 4 B/lane × 32 lanes = 128 B transaction. **Coalesced.**

### Leaf 5: block_table global lookup
- **Depth: 0** (arithmetic — loaded once per CTA per page straddled).
- Address: `addr(b, page_id) = bt_base + b*max_num_pages + page_id`.
- Loaded cooperatively into a small SMEM vector at CTA entry (one or a few int32s per CTA).

### Leaf 6: seq_lens[b]
- **Depth: 0**. Loaded once per CTA; single int32; lives in an SMEM slot or a warp-broadcast register.

**Chain-depth audit:** the deepest chain is Leaf 2 at depth 1, identical to Tier 2. **No new FSM micro-phases added beyond Tier 2's page-table lookup → K gather → MMA** pattern. File 09 is consistent with D1's "no new GATE structure."

## SMEM tile sizing (MEA pre-check)

For a CTA grid `(ceil(N_max / MMA_N), B)`:

| Buffer | Size formula | MMA_N=64 | MMA_N=128 | MMA_N=256 |
|---|---|---|---|---|
| Q_smem (FP8, [64, 128]) | 64 × 128 B | 8 KiB | 8 KiB | 8 KiB |
| K_smem (FP8, [MMA_N, 128]) | MMA_N × 128 B | 8 KiB | 16 KiB | 32 KiB |
| Scale_smem (FP32, [MMA_N]) | MMA_N × 4 B | 256 B | 512 B | 1 KiB |
| Scores_smem (FP32, [64, MMA_N]) | 64 × MMA_N × 4 B | 16 KiB | 32 KiB | 64 KiB |
| mbarriers (u64 each × ~4) | 32 B | 32 B | 32 B | 32 B |
| block_table slice | 4 × pages_in_tile B | 4 B | 8 B | 16 B |
| **Total** | | **~32.3 KiB** | **~56.5 KiB** | **~105 KiB** |

B200 SMEM budget (dynamic SMEM opt-in): up to ~228 KiB per CTA. All three candidates fit. MMA_N=64 and MMA_N=128 both permit ≥2 CTAs per SM; MMA_N=256 likely drops to 1 CTA per SM and reduces occupancy.

**D3 will choose between MMA_N=64 (max occupancy, N-tile = 1 page — no straddling) and MMA_N=128 (half the launch count, 2-page straddling).** Measure decides.

## Combine Group (REL)

File 09: Combine Group = "relation between two REDUCE dimensions whose outputs feed the same final value." Here D is the only REDUCE dim; the 4 K-slabs are partitions of D, not separate dimensions. File 03 Matrix cell (REL × SRG): "between two SRG reductions — inter-reduction dependency." The 4 slabs' merge is the canonical SRG×FXP Online Algorithm, not a Combine Group across independent dims. Formally: no Combine Group is introduced by this delta; the slab merge is a fixed-point accumulation within one reduction dim.

**REDUCE_MERGE phase is NOT generated at Step 3** — the MMA hardware performs the merge transparently via its accumulator register (TMEM with enable_input_d=true).
