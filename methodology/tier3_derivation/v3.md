### ⚠ Note 1 — Phase tracking for mbarrier across pages

The mbarrier is reused across PAGES_PER_CTA=4 page iterations. Each iteration: commit → try_wait → ld. Per gau-nernst `mbarrier_wait(mbar_addr, phase)`, the `phase` flips each iteration. D3 does not explicitly initialize phase to 0 or flip it — **D4 must specify** `phase` initialization and flip logic per the GT-2 reference pattern.

Action: D4 must add `int phase = 0; ... mbarrier_wait(mbar, phase); phase ^= 1;` inside the page loop.

### ⚠ Note 2 — Warp-0 vs warp-1 for MMA issue

D3 Decision 8's code sketch has `if (warp_id == 0 && elect_sync())` for MMA issue, while GT-2 says the allocator is warp 1 (warp_id==1). Either warp can issue MMA via elect_one_sync. D4 must pick ONE warp explicitly. **Action: D4 will fix MMA issue to warp 0 (elected thread), TMEM alloc to warp 1, mbarrier init to warp 0 thread 0 — consistent with gau-nernst pattern.**

### ⚠ Note 3 — K-slab loop MMA sequencing

D3's MMA sketch issues 4 `tcgen05.mma` instructions in a loop without any `tcgen05.fence::before_thread_sync` between them. The PTX ISA allows multiple in-flight MMA instructions; the `commit` after the 4th completes all prior MMAs. **Per GT-2 and tcgen05_sync_fences.md, this is correct — fence is only needed around the mbarrier wait, not between MMAs within the same warp's issue stream.** No action needed.
